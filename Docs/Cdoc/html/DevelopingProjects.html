Naming Conventions

Controller and Delegate Classes

  • Apple's recommendation is to name these classes <Name>WindowController, <Name>ApplicationDelegate, and so on. We tried it, and these names are not only long to type, they are frequently enough abbreviated on display to be accessibility and convenience issue, and all the extra verbiage gets in the way of quickly recognizing what the classes are for.
  • Application delegate classes: name <Name>App
  • Window and View Controller classes: name <Name>WindowC, <Name>ViewC, <Name>SheetC and so on.

Test Types and Suffixes

Unit Tests (*_UTest)
test unit and close collaborators only; other collaborators are mocked or otherwise substituted for
Functional Tests (*_FTest)
test functionality of a group of related classes
Acceptance Tests(*_ATest)
verify that product's functional requirements are met
Integration Tests (*_ITest)
test scenarios of the software (eg. end-to-end)
Performance Tests (*_PTest)
use comparative timing to track performance of operations
Xcode Tests (*_XTest)
specialized Xcode target type to test aspects of a running Cocoa application

Porting C++ to Objective-C (OLD)

Porting Overview

  • Motivation
    • For Mac development, C++ is being replaced by Cocoa with Objective-C.
    • Xcode supports and we consistently use a testing framework for Objective-C, but not for C++.
    • C++ code is incompatible with Objective-C Automatic Reference Counting.
  • AccessorC9
    • These techniques are intended for use in porting C++ code from AccessorC9 to its Cocoa version Accessor.
      • We exploit some characteristics of AccessorC9 in this process.
      • AccessorC9 code is contained in namespace AccessLib, alias al.
        • ObjC objects <ClassCP> are used to wrap C++ objects <al::Class>.
      • AccessorC9 class al::Host was intended to support polymorphism for executable type and context.
        • al::Host became bottleneck access point to the Application, Window, and SlateManager classes.
        • We now exploit those aspects to provide an access to application-level Objective-C classes from C++ code.
  • As much as possible, we will wrap rather than change existing code until the initial import is complete.
Porting: Naming Conventions
  • Original C++ files for class <Class> are named <Class>_AL.cp/h (or <Class>_C9.cp/h). These are initially imported as is.
    • The implementation file may later be renamed to use extension .mm to enable use of Objective-C code.
  • Files for an Objective-C class that wraps the class in files <Class>_AL.cp/h are named <Class>.mm/h.
    • The class itself is temporarily named <Class>CP, since otherwise the compiler could sometimes confuse <Class> with al::<Class>.
  • Pure Objective-C classes are in files named <Class>.m/h. Ultimately, pure Objective-C classes will replace all C++ and Objective-C++ classes.
