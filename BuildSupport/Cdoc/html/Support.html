CCDev

CCDev is a system for supporting our development work. It includes:

  • folder CCDev containing scripts and other supporting functionality
    • all files in folder CCDev are derived
    • environment variable $CCDev contains the path to folder CCDev
  • target CCDev, which generates much of the content of folder CCDev

Scripting

Development Environment

CCDev supports scripting in both Unix and AppleScript.

Effective January 3, 2011, we use the Korn shell for Unix development. This decision was made on the advice of Scott Clark, and has been working very well.

ccInstall

CCDev supports an installation system used to process script-based targets. It is used primarily to install technical documentation, application configuration automation support, and general developer scripting support.

Install Scripts

Install scripts are files named <Name>_install.ksh, and are normally stored at the root level of their parent folder. Each script processes the files in its folder.

Install scripts are run in place, with the current directory set to their immediate parent folder. Any scripts or functions called should be invoked from their installed locations.

A single parameter specifies the command(s) to be executed:

clean
trigger files for rebuilding; may perform additional cleanup
install
install files changed since last build, or all files if target was cleaned
test
execute shunit test files _Tests/test*.ksh associated with this target
upload
copy files from their installed locations to server

ccInstall Script

ccInstall provides support for install scripts.

Cdoc (technical documentation) targets can normally be installed directly by ccInstall, instead of requiring a script Cdoc_install.ksh. If the Cdoc folder contains a script <ProjectName>Additions.ksh, it will be executed as part of the install process.

Source Control

A Source Control Management system supports tracking the source files used in a project at multiple stages of development. It maintains a repository of the files, and updates that repository when the user requests that the files be "checked in". It normally stores only the differences between successive copies of a file, which saves space and, for text files, supports viewing any two revisions of the same file side-by-side.

The Source Control system currently used at C & C software is Subversion, which, among other advantages, does use binary comparison when handling binary files.

Source Control Policies

  • WARNING: Do not attempt to rename a folder and change its contents on the same revision. This does not work in Subversion, and attempting to do so apparently can corrupt the repository.
  • CAUTION: Prior to Xcode 4, we always used BBEdit or Terminal to add items to a repository because Xcode 3 did not respect svn:ignore or [auto-props] for file types. This has not yet been checked in Xcode 4.

Repository Backup

  • Status 30 May 2011
    • For some time, Subversion repositories have been backed up with an automatic MobileMe Backup scheduled daily to an external hard drive and to iDisk
    • Recently, in-house script ccBackup was added. This updates a mirror of selected folders including the Subversion repository to an external hard drive on demand.
    • However, copying a Subversion repository may generate a unusable repository if another user accesses the repository during the copy process. This becomes a problem now, as other users start to use repositories. Subversion provides a hot copy option/script that effectively deals with this problem.
    • Also, Subversion supports dumping repository contents to a text file in full and incremental versions. We have not used this in the past, but should implement it now.
  • Repositories Backed Up as Follows:
    • Backup with an automatic MobileMe Backup scheduled daily to an external hard drive and to iDisk. These backups are subject to corruption if another user accesses the repository while the backup is in process, but with the low level of remote traffic we expect, this is expected to be rare. Most backups are incremental, so the previous backup would still be available.
    • Subversion hook script runs Subversion's hotcopy (via Subversion's hot-backup.py script) whenever a new revision is created.
    • ccBackup (in-house script, run on demand) maintains hot-copied repositories with its disk image.
    • scheduled launchd tasks maintain repository dumps, with full dumps weekly and incremental dumps daily

File Types

  • Human-Edited Text Files
    • These are the ideal type of file to use with version control system because:
      • different versions of a file can easily be compared side-by-side, with differences highlighted
      • files are also stored by tracking only the differences, which allows storage of a lot of information in a compact form
      • Normally, all files of this type are versioned; we know of no exceptions at this time.
  • Generated Text Files
    • similar to human-edited text files; generated by many programs, frequently in XML
    • may be large; may contain frequent and spurious changes; may not be human-readable
    • some types, currently several that track user status (eg. current selection, window size) for individual files, are excluded
  • Binary Files
    • user cannot easily compare different versions of a file
    • essential binary files, as determined by the needs of each project, are versioned
iWork

iWork documents are in general incompatible with Subversion. When iWork resaves any file, it overwrites any existing content, including the contained subversion data. The workarounds that I have been able to find appear to be quite complex and probably impractical for any of the uses we currently envision for iWork files.

Numbers files can be exported to .csv or .xls formats. Pages files can be exported to plain text or RTF. We do not currently foresee a need for Keynote files. Needing to reexport before check in is an inconvenience and is vulnerable to human error.

Conclusions:

  • avoid using iWork files in development projects
  • if iWork files are used, provide a script that can be run at check-in time to prompt the user if any affected exported files are older than their sources
previous iWork notes
  • we currently have experience with Pages and Numbers documents, which share a common package structure:
    • Contents/PkgInfo ("Document", apparently binary)
    • QuickLook/Thumbnail.jpg
    • document-thumbnail.tiff (Numbers only)
    • index.xml.gz
      • binary archive contains single .xml file, not really human-readable
      • more complex files might have more than one such archived files
  • the fact that text files in these packages are zipped is an inconvenience for source control that might or might not be worth the trouble to work around
  • one possible way to handle versioning for these files might be:
    • "building" an iWork file would:
      • create a sibling folder with the same base file name; copy PkgInfo, any .xml.gz files, and any graphics files with possible exclusions for thumbnails, to that folder
      • in the new folder, unzip any zipped files and remove their originals
    • have the source control system ignore the iWork package but check in the sibling folder
    • "building" the sibling folder would reconstruct the iWork file

Repository Access

Repository Access Tasks`

Checkout Repository

This will place a complete set of all the active files in the repository at a location you specify on your hard drive. This set of files is called a "working copy".

  • You need a separate folder on your hard drive to hold working copies for multiple projects. If you do not already have a working copy folder, create one, probably either inside your Documents folder or directly in your home directory. Mine is called "Dev", and is in my home folder.
  • Determine the complete pathname for a repository folder inside your working copy folder. The path should use start with "${HOME}" to specify your home directory, and should end with the name of the repository. Examples: "${HOME}/Dev/Punkin"; "${HOME}/Documents/Projects/Support"
  • Terminal: svn checkout svn://<IPAddress>/<repos>/trunk <workingCopyPath>/<repos>
    • In this and similar instructions, any expression like <aLabel> is to be replaced by whatever aLabel represents. For example:
      • <repos> stands for the name of the repository you are checking out
      • <IPAddress> stands for the IP address of the server that is the host for the repository
      • <workingCopyPath> stands for the path to your working folder, starting with ${HOME}/
    • It is important to type this command exactly. In particular, omitting "/trunk" from the repository address or the repository name from the <workingCopyPath> are easy mistakes to make that will have undesired results.
    • If Terminal says "Can't connect to host ... Connection refused", then the server is probably offline; contact administrator.
    • When asked for your username and/or password in this or another subversion operation, and to your subversion username and password.
  • Verify that there is now be a folder named <repos> in your working copy folder that contains files already in the project.
  • Continue with Xcode: Add Repository below.
Xcode: Add Repository

Tell Xcode about an existing repository. Then you will be able to use Xcode to manage your work with that repository.

  • Open Xcode; select Window > Organizer; from the set of five tabs at the top of the Organizer window, select "Repositories".
  • The Repositories panel on the left shows a list of the repositories Xcode knows about. It is initially empty.
  • Add repository named <repos> to Xcode repositories:
    • Click the "+" button at the bottom of the Repositories panel; choose "Add Working Copy..." from the popup menu. Choose the folder to be added from inside your working copy folder.
    • The repository now appears in your repositories list. Select the repository -- it is called, <repos>, and it has a brown repository icon on the left and the green or yellow status dot on the right. If the status dot is yellow, type your Subversion username and password into the fields provided; the yellow dot should turn green to tell you that the repository can now be accessed.
    • Click the disclosure triangle next to the repository if it is not already open. The purple folder "root" represents the repository. Clicking on the purple folder shows the files in the repository; your working copy files are in folder "trunk". The blue folder represents your working copy; clicking on it displays the files in your working copy. The bottom of the screen shows the repository history -- what, when, why and by whom changes have been made. Once you've checked something in, your comments will appear here.

    • Advantages
      • Cost: this host is free to us because we had an existing account; other supposedly free web hosting services have significant disadvantages such as problematic advertising requirements
      • Installation Convenience: I can mount my iDisk and access the files very conveniently
    • Advantage or Disadvantage?
      • Does not support FTP (no wonder I was having trouble!). MobileMe is easier for me to script now, but the technology is non-standard and therefore not compatible with what we would need with a different host.
    • Disadvantages:
      • Does not support server-side technologies such as PHP, so there's no provision for communicating anything back to us other than via e-mail, which is standard HTML technology.
      • Acceptable password protection requires server-side technology which, in the case of MobileMe, means that we must rely on the hosting service to provide password protection. But the MobileMe service provides password protection only for iWeb sites (per MobileMe support 3/28/11).
      • Therefore, the only way I could find to get password protection for our website was to put it on my iDisk at a location other than either Sites folder. This may turn out to be less convenient.
  • Client and Server Code (as I now understand it):
    • Client-side code is HTML, CSS, and JavaScript that makes up the website. Client-side code is limited to text files that can potentially be read by the client.
    • Server-side code is that connected with the server that responds to requests to provide webpages. Server-side code can be in scripts in languages like PHP and Perl, but can also access code in other forms. Only server-side code can communicate with the provider of the website.

File Server

We need a file server to allow remote access to the Subversion repositories by multiple developers. We may also at some point wish to use it to serve webpages.

There are two servers supported for use with subversion:

  • svnserve
    • specialized server provided with Subversion
    • simpler to set up that Apache
  • Apache
    • widely used general-purpose server
    • supports more features and options than svnserve; comparative strength and getting through firewalls may be relevant to us

See also Tools: Apache 2.

Apache Security Configuration

  • We went to a great deal of trouble to figure out what security precautions to take to protect our server and the files it serves, and to keep the server on for quick testing only until that was done. In the end, we had decided on precautions that almost exactly matched what had already been done in the default installation we had used (MacPorts).
  • File Permissions
    • directory /opt/local/apache2 is owned by root:admin [default]
    • in general, directories have permissions 755 (read-only except for owner; files can be added by all) [default]
    • in general, files have permissions 644 (read-write for owner, read-only for others); execute permissions are added (755) for executable files [default]
    • logs and configuration information are readable only by owner [custom]:
      • chmod -R go-r /opt/local/apache2/conf
      • chmod -R go-r /opt/local/apache2/logs
  • configuration file httpd.conf
    • set Listen to <IPAddress>:80 [asInstructed]
    • set ServerName to svn.candcsoft.com:80 (or <IPAddress>:80 if domain name is svn.candcsoft.com not configured) [asInstructed]
    • set restrictive rules for the default Directory configuration; rules can then be overridden for specific directories
      • specify that no files are to be served to anyone [default]
      • allow following of symbolic links [default]
        • According to the Apache documentation: restricting symbolic links doesn't really work as a security precaution because it can be gotten around; following symbolic link does not change the Directory rules; restricting symbolic links can cause performance problems
      • do not allow .htaccess files, which can be placed under the web server tree to affect configuration [default]
    • do not allow access to Web server's document root [custom]
      • Apache allows this by default. But it exposes the websites at ~/Sites, which contains our local copy of the Punkin documentation. (Turning Web Sharing on also exposes ~/Sites).
      • comment out the active statements in the Directory /var/www/htdocs directive

Permissions Policy

In general, we use default permissions for most filetypes of rwxr-xr-x for directories and rw-r--r-- for files.

Files, directories and applications using different permissions:

  • Scripts and script templates, except Javascript: rwxr-xr-x
  • iWeb uses rwx------ for everything

CCDev_Setup

CCDev.setup is a Unix script that sets up our development environment. Script is run prior to attempting to build any other targets or do other development work. It can be run from Terminal or by building Xcode target Support:CCDev_Setup.

CCDev.setup is available for collaborators to download. For all collaborators, script sets up basic environment including git support. For administrator, it also installs additional environment files, third-party tools, and starter scripts required for CCDev.install to run.

Normally, CCDev.setup is run once only, with subsequent maintenance as files change handled by CCDev.install. CCDev.setup does not avoid overwriting existing files.

Notes
Reference Files

Reference Files are read-only copies of external files, usually from another repository, that are included in a workspace for reference. Reference filenames are prefixed with "_rf_" to allow git to ignore them.

  • Files not included in a workspace cannot be displayed by Assistant.
  • References to external files can be placed in a workspace. However, then Xcode knows about them and tries to support git operations with their projects, which makes Merge, at least, more complicated and much more error-prone.
